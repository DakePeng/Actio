# Hot Update Design — Actio Desktop

**Date:** 2026-04-18
**Scope:** Auto-update mechanism for the Actio Tauri desktop app, Windows-only (x86_64) for the initial rollout.

## Goals

- Users automatically receive new versions without manual downloads.
- Both frontend and backend update together in a single atomic bundle.
- Distribution uses GitHub Releases as the file server (public repo).
- Updates are cryptographically signed; the client refuses tampered or unsigned bundles.
- Release process is CI-driven: pushing a version tag produces a signed release with no manual steps.

## Non-goals

- Cross-platform support (macOS/Linux) — deferred. The design leaves room for it via tauri-action's built-in multi-platform matrix.
- Decoupled frontend-only or backend-only updates. Frontend is bundled into the installer; a FE-only change still cuts a full release.
- Delta/patch updates. Each release ships the full installer (~tens of MB expected). Revisit if install size grows past the point where this matters.
- Multiple channels (beta/stable). Single `latest.json` stream. Channels are a future extension.
- Rollback-in-place. If a bad release ships, recovery is "cut a higher-versioned fix release." The bad release can be deleted from GitHub to stop new clients from pulling it.

## Architecture

Three pieces:

### 1. Client (Tauri app)

- `tauri-plugin-updater` handles check / download / signature verify / install.
- `tauri-plugin-process` provides relaunch capability (required by the updater).
- On app startup, a background task calls `updater().check()`. If an update is available, it calls `update.download_and_install()`. The plugin verifies the Ed25519 signature against the baked-in public key, runs the NSIS installer in `passive` mode (visible progress bar, no prompts), closes the app, installs, relaunches into the new version.
- Network failures, signature mismatches, and corrupted downloads are logged and swallowed — the app continues to run on the current version. No user-facing error dialogs.

### 2. GitHub Release (file server)

Each tagged release carries:

- `Actio_<version>_x64-setup.exe` — NSIS installer (Tauri's default Windows bundler output)
- `Actio_<version>_x64-setup.exe.sig` — Ed25519 detached signature
- `latest.json` — update manifest auto-generated by `tauri-action`, schema:

```json
{
  "version": "0.2.0",
  "notes": "Release notes from the GitHub release body",
  "pub_date": "2026-04-18T00:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "<base64 Ed25519 signature>",
      "url": "https://github.com/<owner>/<repo>/releases/download/v0.2.0/Actio_0.2.0_x64-setup.exe"
    }
  }
}
```

The client's updater endpoint is `https://github.com/<owner>/<repo>/releases/latest/download/latest.json`, which GitHub resolves to the latest non-prerelease release's `latest.json`.

### 3. CI release workflow

`.github/workflows/release.yml` — triggered on tag push matching `v*.*.*`. Uses `tauri-apps/tauri-action@v0` to build, sign, and publish in one step. Secrets: `TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`.

## Components in detail

### Cargo dependencies (`backend/src-tauri/Cargo.toml`)

```toml
tauri-plugin-updater = "2"
tauri-plugin-process = "2"
```

### `tauri.conf.json`

```json
"plugins": {
  "updater": {
    "active": true,
    "endpoints": [
      "https://github.com/<owner>/<repo>/releases/latest/download/latest.json"
    ],
    "pubkey": "<ed25519 public key>",
    "windows": { "installMode": "passive" }
  }
}
```

Capabilities (`backend/src-tauri/capabilities/default.json` or equivalent) must grant `updater:default` and `process:allow-restart` to the main window.

### `lib.rs` setup hook

```rust
.plugin(tauri_plugin_updater::Builder::new().build())
.plugin(tauri_plugin_process::init())
.setup(|app| {
    let handle = app.handle().clone();
    tauri::async_runtime::spawn(async move {
        use tauri_plugin_updater::UpdaterExt;
        match handle.updater() {
            Ok(updater) => match updater.check().await {
                Ok(Some(update)) => {
                    if let Err(e) = update
                        .download_and_install(|_chunk, _total| {}, || {})
                        .await
                    {
                        tracing::warn!(error = %e, "update install failed");
                    }
                }
                Ok(None) => tracing::debug!("no update available"),
                Err(e) => tracing::warn!(error = %e, "update check failed"),
            },
            Err(e) => tracing::warn!(error = %e, "updater unavailable"),
        }
    });
    Ok(())
})
```

The check runs detached — it does not block app startup. If the user interacts with the app before the download finishes, that's fine; the installer only fires once download completes, at which point the app is closed and relaunched.

### CI workflow (`.github/workflows/release.yml`)

```yaml
name: Release
on:
  push:
    tags: ['v*.*.*']

jobs:
  release:
    runs-on: windows-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'pnpm' }
      - uses: dtolnay/rust-toolchain@stable
      - run: pnpm install --frozen-lockfile
        working-directory: frontend
      - uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
          TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
        with:
          projectPath: backend/src-tauri
          tagName: ${{ github.ref_name }}
          releaseName: 'Actio ${{ github.ref_name }}'
          releaseDraft: false
          prerelease: false
          includeUpdaterJson: true
          args: '--target x86_64-pc-windows-msvc'
```

## Data flow

**Release cut:**

1. Dev bumps version in `backend/src-tauri/tauri.conf.json`, `backend/src-tauri/Cargo.toml`, `frontend/package.json`. All three must match.
2. Dev commits the version bump, tags `v<version>`, pushes tag.
3. CI workflow runs: `pnpm install` → `pnpm tauri build` (signs bundle with the private-key secret) → creates GitHub release → uploads `.exe`, `.exe.sig`, `latest.json`.

**Client update cycle:**

1. App launches, spawns background update-check task.
2. Task GETs `https://github.com/<owner>/<repo>/releases/latest/download/latest.json`.
3. Plugin compares the manifest's `version` to the app's running version (semver). If newer, proceeds; if same or older, stops.
4. Plugin GETs the installer `.exe`, verifies the Ed25519 signature against the manifest's `signature` field using the baked-in pubkey.
5. Plugin spawns the installer with `/P` (passive) flag. NSIS shows progress bar, closes the running app, replaces the binary + resources (including sherpa-onnx DLLs), relaunches.

## Error handling

| Scenario | Behavior |
|---|---|
| No network at launch | `check()` returns error, logged as warning, app continues. |
| `latest.json` returns 404 | Same as above (e.g., if a release was deleted and no newer one exists). |
| Manifest parse error | Logged, update attempt skipped. |
| Signature verification fails | Plugin refuses to install, logs error. This is the critical security boundary — a compromised GitHub account cannot push malware. |
| Download interrupted | Plugin reports error, logged. User retries next launch. |
| Installer fails mid-run | User can manually download from the GitHub releases page. `RELEASING.md` documents this path. |
| Downgrade attempt (older version in manifest) | Plugin compares semver and refuses. |
| User has no write permission to install dir | NSIS fails; falls back to current version. User needs to reinstall from a fresh download. |

## Preserved state across updates

- **ASR models** live in `AppLocalData`, outside the installer. Untouched by updates.
- **`settings.db`** (sqlite, app config + history) lives in `AppLocalData`. Untouched.
- **Speaker enrollments / voiceprints** live in `settings.db`. Untouched.
- **sherpa-onnx DLLs** are bundled as installer resources and get replaced atomically with the rest of the app.
- **Registered global shortcuts** are re-registered on relaunch by existing startup code.

Any schema migrations needed by a new version are handled by the app's existing sqlx migration code on the next startup. Updates do not themselves run migrations.

## Testing plan

1. **Signature enforcement** — build v0.0.1 with pubkey-A baked in; build v0.0.2 signed with key-B (mismatched); confirm v0.0.1 refuses to install v0.0.2 and logs a signature error.
2. **End-to-end staging** — cut a `v0.0.0-e2e.1` pre-release with a visible change (e.g., window title text). Install on a clean Windows VM. Push `v0.0.0-e2e.2`. Launch app, confirm it downloads, closes, installs, and relaunches showing the new title.
3. **Offline resilience** — disable network before launch. Confirm app starts normally, no hang, no error dialog. One warning in logs.
4. **Downgrade refusal** — with a client on v0.0.2, publish a `latest.json` pointing at v0.0.1. Confirm client does not downgrade.
5. **Settings/DB preservation** — across a real update, confirm `settings.db`, downloaded ASR models, and speaker enrollments all survive.
6. **Manual recovery path** — simulate installer failure; document and verify that reinstalling from the GitHub releases page works.

## One-time setup tasks

1. `pnpm tauri signer generate -w ~/.tauri/actio.key` — generates Ed25519 keypair (password-protected private key).
2. Copy public key into `tauri.conf.json`'s `plugins.updater.pubkey`.
3. Add private key file contents as repo secret `TAURI_SIGNING_PRIVATE_KEY`, password as `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`.
4. **Back up the private key** to a password manager offline. Loss of the private key means no existing installation can ever receive another update — they would all have to be manually reinstalled from scratch with a new pubkey.
5. Document the release flow in `RELEASING.md`.

## Rollout checklist

1. Generate signing keypair; commit pubkey to `tauri.conf.json`.
2. Add plugin dependencies, setup hook, capability entries.
3. Bump version to `v0.1.1` (first updatable version).
4. Add `release.yml` workflow; configure repo secrets.
5. Tag `v0.1.1`, push, verify a release appears with `.exe`, `.exe.sig`, `latest.json`.
6. On a clean Windows VM, install a hand-built `v0.1.0`, then push `v0.1.2`, and verify auto-update kicks in on next launch.
7. Write `RELEASING.md` covering: version-bump locations, tag format, recovery from bad releases, private-key custody.

## Future extensions

- **Cross-platform:** add macOS + Linux targets to the tauri-action matrix once signing/notarization (macOS) and AppImage packaging (Linux) are sorted.
- **Channels:** split into `latest.json` (stable) + `latest-beta.json` (beta). Expose a channel selector in settings.
- **Update UX upgrade:** switch from immediate install to "notify-and-defer" once user feedback suggests the current flow is interrupting sessions. Plugin supports pause/resume hooks.
- **Delta updates:** only worth adding if installer size becomes a distribution problem.
